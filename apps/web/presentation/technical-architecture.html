The system, as built

How Haggl actually works

Haggl reads a hospital bill, finds the billing errors and the statutes on the patient's side, then places a real phone call and talks the balance down. The language model is the mouth. The deterministic code is the brain. Every number the agent speaks on the line traces to a database query or a config value, and the code, not the model, decides what is true.

a.

Runtime topology

Who talks to whom when a patient runs a case

The browser is served from hagglfor.me behind Cloudflare. It talks to Next.js (the six screens and the War Room), which calls the FastAPI engine. FastAPI owns the deterministic pieces: flag detection, the strategy dossier, the ladder state machine, the mid-call tools, and the post-call webhooks. Supabase is the spine underneath: Postgres for cases, calls, call events and outcomes, Storage for uploaded documents and call recordings, and Realtime, which streams every call_events insert straight into the live War Room.

The voice itself lives at ElevenLabs: the negotiator agent (it speaks as "Alex," runs on a gpt-5.4 brain, and uses the Jason voice) plus four counterparty personas for the simulated market. Calls go out over Twilio's native PSTN integration to a real phone. Mid-call, the agent reaches back into FastAPI through webhook tools for every fact and every move, and after the call the transcript and audio flow back through the post-call webhook.

b.

One negotiation turn

Where the LLM fires, and where it does not

Every exchange on a live call runs the same loop. The rep says something, the agent proposes a move in words, and then it hands control to the server: it calls report_lever_result with what just happened, and the deterministic advance() in the ladder state machine decides the next move. The agent voices whatever comes back. It never picks the escalation, and it never chooses the price.

engine/state_machine.py · advance()

Every reported result runs this gauntlet before any move is allowed. Each gate is plain, tested Python, not a prompt instruction.

  1. 1Bounds check. A non-positive offer, a plan under 1 or over 96 months, or interest outside 0 to 100 percent is rejected outright, so a garbled report can never bank nonsense.
  2. 2Floor check. Any offer above dossier.floor (the most the patient can actually pay) is rejected before it can be spoken.
  3. 3Plan total, with interest. Payment plans are multiplied out as monthly × months × (1 + interest). A plan that only clears the ceiling once interest is added is still rejected, and the note names the interest as the culprit.
  4. 4Repetition cap. The same non-accepted point made three times in a row is parked as an open item, and the call moves to the next lever instead of grinding.
  5. 5Question-coverage gate. The agent cannot walk off a rung with required questions (for example account hold, records alignment, rep name) until those tags are covered. One block per rung, so a live call is never hard-deadlocked.
  6. 6Bad-faith, two strikes. A stonewall parks the topic rather than escalating; reaching a supervisor is a last resort, and forced escalations are capped at two per call before the agent closes out with a documented decline.

Settling above dossier.target is allowed, but only with an explicit escalation flag that surfaces in the War Room as a human-approval event. That is exactly why the flagship demo settles at $1,650: under Maya's $1,700 ceiling, so allowed, but above the engine's target, so a human confirmed it on camera.

c.

The honesty layer

The agent cannot bluff, by construction

Numbers reach the agent only through tools. get_case_brief returns the verbatim case file. get_benchmark is the single citable price source. get_authorization returns the patient's recorded consent. The config names the allowed sources in honesty.citable_sources_only, the deployed tool descriptions repeat the rule to the model, and if the agent did not get a number from a tool or from the case brief, it does not say it.

Verified, not asserted

After every call, a deterministic audit reads the full transcript against a per-case set of allowed numbers and produces a pass or fail that lands in the War Room. The thing that judges the AI is itself not an AI: it is plain regex and arithmetic in engine/honesty.py.

D1 · disclosure

Says it is an AI

The agent discloses that it is an AI acting for the patient. Under the team's disclose-when-asked policy, this is satisfied by the robot-question check below.

D2 · never deny

Never denies being AI

If the rep asks whether it is a bot, the agent confirms plainly and immediately. Denial is an automatic fail. The config sets never_deny_ai: true as absolute.

D3 · number honesty

Every number traces

Each dollar figure the agent speaks must match the allowed set or a number the counterparty said. An uncited number fails the audit.

Two more boundaries hold on the line. Prompt injection is refused by policy: the rep is never the agent's operator, so a request to reveal instructions or to dictate what a tool returns is declined once, politely, and treated as bad faith on repeat. And when a rep challenges the caller's authority, the recorded patient authorization is presented verbatim: the agent reads the exact words on file, states plainly that it is reading a recording rather than playing audio on the line, and never calls a recording a signed release. A separate audit catches that overclaim if it ever happens.

Why this layer exists

The first tool-less test call invented a CPT code and a date out loud. The tools, the citable-source rule, and the post-call audit are the direct answer to that failure. Code decides what is true; the model decides how to say it.

d.

The data layer

Real prices, real statutes, computed keys

The benchmarks the agent cites are grounded in real market data. The chargemaster is 881,668 per-payer rows pulled from three Boston hospitals' published price-transparency files (Massachusetts General, Brigham and Women's, and Newton-Wellesley), staged into Postgres and queried per code. Alongside it sits real CMS Medicare data: PFS rates computed from RVUs and Boston-locality GPCIs, OPPS Addendum B for hospital outpatient facility fees, and the CLFS for labs, all staged by code with their formulas attached.

The multi-anchor engine (engine/anchors.py) builds an anchor set for every bill line. Each anchor carries full provenance: a source, a source URL, the formula, a confidence level, and a display label. A missing lookup becomes an honest coverage status, never a guessed price.

Medicare rate RVU × GPCI × CF Hospital cash price posted MRF Cross-payer band p25 · median · p75 RAND-norm estimate ×2.54, labeled

Nine scenarios (a duplicate, an upcoded ER visit, an unbundled panel, a self-pay gross bill, an EOB mismatch, an out-of-network balance bill, and more) ship with answer keys generated by code, not by hand. Tests run the whole pipeline over each scenario's artifacts and assert exact equality with its key, so the numbers cannot drift. The flagship arc, all of it computed and reconciled by transform.py --check:

$4,287 billed → $3,875 (duplicate removed) → Medicare $438 · cash $2,633.25 cited → $1,650 settled, -62%

Provenance kept honest: the demo locks the $438 Medicare total for consistency with the rehearsed arc and the archived call audio. The real CMS MA-01 Boston rates for those exact lines total $1,373.79 and live in data/seed/medicare_rates.json, with the code-computed key at data/scenarios/sc01_maya_baseline/answer_key.json. The story is checkable, not pending.

f.

Config, not code

One YAML per market

Everything specific to medical bills is data in a single file, config/verticals/medical_bills.yaml. The engine modules read that file and hard-code none of it. A moving.yaml stub sits next to it against the same schema, which is the honest measure of the claim: retargeting the system is writing that file, plus the benchmark data for the new market. Changing a vertical touches exactly these knobs, and no engine code:

  • Red-flag thresholds and the E/M upcode pairs
  • Both negotiation ladders, provider and collections
  • Fair-band multiples over Medicare
  • Routing thresholds and the charity-care gate
  • Required questions per rung
  • Escalation policy and the stonewall trigger phrases
  • Persona roster for the simulated market
  • Voice settings: model, voice ID, speed, brain LLM
  • Disclosure policy and the opener
  • Honesty boundary: the citable sources list

The rule the whole build follows: anything with a number, a threshold, or a legal claim is computed in code and served to the model through a tool. The model decides how to say it. Code decides what is true.
